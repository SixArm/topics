# Static Application Security Testing (SAST)

Static Application Security Testing (SAST) is a white-box testing methodology that analyzes application source code, bytecode, or binaries for security vulnerabilities while the application is not running. It inspects code from the inside out to find potential flaws before software is compiled or deployed.

SAST tools work through several complementary techniques. Code parsing breaks down source code into an intermediate format, often an Abstract Syntax Tree, to understand structure and logic. Taint analysis tracks the flow of untrusted data from sources like user input to sensitive functions, checking whether data is properly sanitized. Pattern matching compares code against databases of known insecure patterns such as the OWASP Top 10 or CWE Top 25. Control flow analysis examines the order of operations to identify logical flaws or execution paths that could lead to vulnerabilities.

The primary advantage of SAST is early detection. Integrating it into an IDE or CI/CD pipeline lets developers shift left, catching and fixing bugs during the coding phase when they are far cheaper to resolve. SAST provides exact file names and line numbers for each finding, along with remediation guidance. It can also analyze the entire codebase, including rarely executed paths that runtime testing would miss. Common findings include SQL injection, cross-site scripting, buffer overflows, hardcoded credentials, and weak cryptographic algorithms.

SAST has notable limitations. It produces false positives by flagging theoretical paths that cannot actually be exploited, which can cause alert fatigue. It lacks runtime context, so it cannot detect server misconfigurations or authentication flaws that only manifest during execution. Tools are also language-specific, so coverage requires selecting the right tool for each language in use. For comprehensive security testing, SAST is best combined with Dynamic Application Security Testing (DAST), which evaluates the running application.
