# Onion Architecture

Onion Architecture is a software design pattern introduced by Jeffrey Palermo in 2008 that organizes an application into concentric layers. Its primary goal is to decouple core business logic from external concerns such as databases, UI frameworks, and third-party services. It shares philosophical roots with Hexagonal Architecture and Clean Architecture, all emphasizing dependency inversion and domain centrality.

The fundamental rule is that dependencies only flow inward. Outer layers may reference inner layers, but inner layers must never depend on anything outside of them. At the center sits the Domain Model, containing business entities, value objects, and invariants that hold true regardless of the technology stack. Surrounding it, the Domain Services layer handles complex business logic that spans multiple entities, such as cross-entity calculations and domain-specific interfaces. The Application Services layer orchestrates workflows by coordinating domain objects to fulfill use cases like "Place Order," but performs no I/O itself. The outermost layer, Infrastructure and Presentation, contains concrete implementations: database adapters, REST APIs, message brokers, and user interfaces.

This layering yields several practical benefits. Sustainability improves because the domain remains independent of infrastructure choices; swapping a SQL database for a document store or changing cloud providers requires no changes to business logic. Testability is straightforward since core business rules can be unit tested without mocking databases or web servers. Long-term maintainability increases because the enforced separation of concerns prevents tight coupling and the gradual decay toward an unstructured codebase.

Onion Architecture works best in applications with significant domain complexity where business rules change independently of technology choices. For simple CRUD applications, the additional layering may introduce unnecessary overhead without proportional benefit.
