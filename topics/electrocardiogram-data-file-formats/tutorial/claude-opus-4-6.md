# Electrocardiogram Data File Formats

Electrocardiogram (ECG/EKG) data file formats define how the electrical activity of the heart is recorded, stored, and exchanged between devices, software systems, and institutions. The landscape of ECG file formats is remarkably fragmented: decades of independent development by medical device manufacturers, standards bodies, and regulatory agencies have produced a diverse ecosystem of proprietary and open formats. For technology professionals building clinical systems, research platforms, or health data integrations, understanding these formats is essential for achieving interoperability, regulatory compliance, and reliable signal fidelity. This tutorial surveys the major ECG data file formats, compares their characteristics, and provides practical guidance for selecting and working with them.

## Why ECG File Formats Matter

An electrocardiogram captures voltage changes over time from electrodes placed on the body, typically producing between 1 and 12 simultaneous signal channels sampled at rates ranging from 250 Hz to 1000 Hz or higher. The resulting data must preserve not only the raw waveform samples but also patient demographics, acquisition parameters, electrode configuration, calibration values, timestamps, and often the results of automated interpretation algorithms. The choice of file format determines how all of this information is structured, how efficiently it can be stored and transmitted, and whether it can be reliably read by systems from different vendors. In practice, format incompatibility is one of the most persistent obstacles in digital cardiology: a recording made on one manufacturer's device may be unreadable on another's without specialized conversion software.

## Anatomy of an ECG Data File

Regardless of format, most ECG data files contain a common set of information organized into a few logical sections:

- **Header or metadata block**: Identifies the patient, recording date and time, acquisition device, sampling rate per channel, calibration parameters (gain, offset, physical units), and electrode configuration.
- **Signal data block**: Contains the digitized waveform samples, either as raw integer values requiring calibration conversion or as pre-scaled floating-point values. Multichannel data may be interleaved (samples from all channels for one time point stored together) or stored channel-by-channel.
- **Annotations and interpretation**: Many formats include fields for automated measurements (QRS duration, QT interval, heart rate), diagnostic statements generated by onboard algorithms, and physician overread notes.
- **Auxiliary data**: Some formats support embedding a rendered graphical report (the familiar ECG strip chart) alongside the digital waveform, enabling both human review and machine reprocessing from a single file.

## Major Open Standards

### EDF and EDF+

The European Data Format (EDF), published in 1992, is a simple, open binary format originally designed for polysomnography but widely adopted for ECG storage in research settings. EDF stores multichannel signals as 16-bit integers with an ASCII header containing all calibration metadata. Its successor, EDF+, added support for annotations, discontinuous recordings, and structured patient identification while maintaining backward compatibility. EDF+ is especially common in Holter monitoring, long-term ambulatory ECG, and academic research, where its simplicity and broad software support make it a pragmatic choice. Its main limitation for ECG use is the 16-bit sample resolution, which may be insufficient for high-fidelity diagnostic applications.

### SCP-ECG

The Standard Communications Protocol for Computer-Assisted Electrocardiography (SCP-ECG) is a European standard (CEN ENV 13734, later ISO 11073-91064) designed specifically for short-duration diagnostic ECG records, particularly the standard resting 12-lead ECG. SCP-ECG uses a compact binary encoding organized into numbered sections, supporting both raw and compressed waveform data (using Huffman coding and bimodal compression), patient demographics, device information, automated measurements, and interpretive statements. It was designed for efficient transmission over low-bandwidth connections and is widely supported by European ECG device manufacturers. SCP-ECG is optimized for the 10-second resting ECG workflow and is less suited to long-duration recordings.

### DICOM-ECG

Digital Imaging and Communications in Medicine (DICOM) is the universal standard for medical imaging. The DICOM Waveform supplement extends this standard to cover ECG data, allowing electrocardiograms to be stored, indexed, and retrieved within Picture Archiving and Communication Systems (PACS) alongside imaging studies. DICOM-ECG encodes waveform data as DICOM objects with full metadata using the DICOM information model, supporting 12-lead resting ECGs, stress tests, and Holter summaries. Its primary advantage is seamless integration into the existing DICOM infrastructure that virtually all hospitals already operate. Its primary disadvantage is the complexity of the DICOM standard itself, which imposes significant implementation overhead compared to simpler formats.

### HL7 aECG

The Health Level Seven Annotated ECG (HL7 aECG) format is an XML-based standard developed through collaboration between HL7 and the U.S. Food and Drug Administration (FDA). It was specifically designed for submitting ECG data in clinical trials, where the FDA requires digital ECG waveforms alongside study data. The aECG format provides rich annotation capabilities, supporting beat-by-beat measurements, interval annotations, and detailed waveform markup. Its XML structure makes it human-readable and amenable to validation against published schemas, but produces significantly larger files than binary formats. HL7 aECG is the required format for FDA ECG submissions and is commonly used in pharmaceutical clinical trials.

### MFER

Medical Waveform Format Encoding Rules (MFER) is an ISO standard (ISO 22077) developed primarily in Japan for encoding medical waveform data including ECG, EEG, and other physiological signals. MFER uses a tag-length-value binary encoding that is compact and extensible. It supports multiple encoding schemes for waveform data and includes provisions for annotations and channel descriptions. MFER has strong adoption in Japan and parts of Asia but limited penetration in North American and European markets.

### ISHNE

The International Society for Holter and Noninvasive Electrocardiography (ISHNE) format is a binary format designed specifically for Holter monitor recordings. It features a fixed-length header with patient and recording metadata followed by raw 16-bit sample data. The ISHNE format is straightforward to parse and has been used as a common interchange format for long-duration ambulatory ECG recordings, particularly in research contexts. Its simplicity is both its strength and its limitation, as it lacks support for annotations and rich metadata.

## Vendor-Proprietary XML Formats

Several major ECG device manufacturers have developed their own XML-based formats for storing and transmitting ECG data. These formats are semi-standardized within each vendor's ecosystem but are not governed by an independent standards body.

- **Philips XML**: Used by Philips PageWriter and IntelliSpace ECG systems. Contains full 12-lead waveform data, automated measurements, and interpretive statements in a structured XML schema.
- **GE MUSE XML**: The export format from GE Healthcare's MUSE ECG management system, one of the most widely deployed ECG databases in hospitals worldwide. Encodes waveforms, measurements, and demographics in a GE-specific XML schema.
- **Schiller XML**: Used by Schiller AG devices, encoding standard and long-duration ECG recordings with metadata and interpretation results.
- **Mortara XML**: The XML format produced by Mortara (now Baxter) ECG systems, commonly found in hospital cardiology departments.

These vendor formats often contain the richest set of metadata and interpretation data for their respective devices, but interoperability between them requires custom parsing and mapping for each schema.

## PDF-ECG

PDF-ECG is a hybrid approach where a PDF document contains both a human-readable graphical ECG strip chart and embedded digital waveform data. The digital data is typically stored as an attachment or in a structured metadata stream within the PDF. This approach allows clinicians to view the ECG using any standard PDF reader while preserving the original digital waveform for reprocessing or quantitative analysis. PDF-ECG has gained traction as a practical format for Electronic Health Record (EHR) integration, since EHR systems universally support PDF documents. The tradeoff is that extracting the embedded waveform data requires format-aware software, and there is no single standard for how the digital data is encoded within the PDF container.

| SCP-ECG | Binary (variable) | Resting 12-lead ECG | Huffman, bimodal | Yes | Yes (ISO) | ~5-20 KB |

## Selecting a Format

The appropriate ECG file format depends on the use case, regulatory context, and system integration requirements. The following guidelines address the most common scenarios:

- **Hospital PACS integration**: Use DICOM-ECG. If ECG data must be stored alongside imaging studies and retrieved through standard DICOM queries, DICOM-ECG is the only format that integrates natively with PACS infrastructure.
- **FDA clinical trial submissions**: Use HL7 aECG. The FDA requires digital ECG data in aECG format for electronic submissions, and the format's annotation capabilities support the beat-level analysis that regulators expect.
- **European diagnostic ECG exchange**: Use SCP-ECG. It is the European standard for resting ECG interchange, offers excellent compression, and is supported by most European device manufacturers.
- **Research data sharing and long-term monitoring**: Use EDF+. Its simplicity, broad tool support, and open specification make it the most practical choice for research datasets, Holter recordings, and multi-center studies.
- **EHR document storage**: Use PDF-ECG. When the primary need is to store an ECG report that clinicians can view in any EHR system, PDF-ECG provides universal readability with optional digital waveform preservation.
- **Vendor-specific workflows**: Vendor XML formats are often the most feature-complete option within a single manufacturer's ecosystem, but plan for format conversion when data must cross vendor boundaries.

## Interoperability Challenges

The proliferation of ECG file formats creates persistent interoperability challenges for technology professionals:

- **Format conversion**: Moving data between systems frequently requires conversion between formats. Lossless conversion is not always possible because formats differ in supported metadata fields, signal resolution, compression schemes, and annotation models. Critical clinical context can be lost in translation.
- **Vendor lock-in**: Proprietary XML formats tightly couple data to specific vendor ecosystems. Extracting data from these systems for use in third-party analytics or research platforms requires reverse-engineering or negotiating access to format documentation.
- **Metadata mapping**: Patient demographics, device identifiers, measurement labels, and interpretation codes are not standardized across formats. Building a universal ECG data pipeline requires defining explicit mapping rules for each source format.
- **Regulatory requirements**: Different jurisdictions and regulatory bodies mandate different formats. A system operating across multiple regions may need to support SCP-ECG for European compliance, HL7 aECG for FDA submissions, and DICOM-ECG for hospital integration simultaneously.

## Content Stored in ECG Files

Beyond the raw waveform, ECG data files frequently carry structured clinical content that varies by format:

| Content Category | Examples | Formats That Commonly Include It |
|---|---|---|
| Patient demographics | Name, ID, date of birth, sex | All formats |
| Acquisition metadata | Device model, sampling rate, filter settings, electrode configuration | All formats |
| Raw waveform data | Digitized voltage samples per channel | All formats |
| Calibration parameters | Gain, offset, physical units, digital range | EDF+, SCP-ECG, DICOM-ECG, MFER |
| Automated measurements | Heart rate, PR interval, QRS duration, QT/QTc interval, axis | SCP-ECG, DICOM-ECG, HL7 aECG, vendor XML |
| Interpretive statements | "Normal sinus rhythm", "Left ventricular hypertrophy" | SCP-ECG, DICOM-ECG, HL7 aECG, vendor XML |
| Beat-level annotations | Individual QRS complex positions, interval measurements per beat | HL7 aECG |
| Graphical report | Rendered ECG strip chart image | PDF-ECG, some vendor XML |

## Summary

Electrocardiogram data file formats span a wide spectrum from simple binary containers like EDF+ and ISHNE to richly annotated XML standards like HL7 aECG and complex medical imaging frameworks like DICOM. No single format dominates all use cases: SCP-ECG excels at compact diagnostic ECG exchange, DICOM-ECG integrates with hospital imaging infrastructure, HL7 aECG satisfies FDA regulatory requirements, EDF+ serves the research community, and vendor-proprietary XML formats provide the deepest feature sets within their respective ecosystems. For technology professionals, the practical challenge is not choosing one format but building systems that can ingest, convert, and faithfully represent ECG data across multiple formats while preserving clinical fidelity, regulatory compliance, and interoperability across institutional and vendor boundaries.
